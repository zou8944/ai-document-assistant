# @文档交互设计

## 背景

用户在聊天时通过 `@` 引用特定文档，期望 AI 仅基于被引用的文档回答。当前前端已完成 mention 解析和 payload 传递，但后端完全忽略了 `document_ids`，导致 @ 无效果。

## 设计原则

**@ =「我在问这个文档」**：用户主动 @ 文档是聚焦行为，AI 应回答基于被 @ 的文档，而非整个知识库。

## 作用域

| 层次 | 操作 | 作用域 |
|---|---|---|
| 会话配置 | 绑定知识库 | 整个会话，持久 |
| 当次对话 | @ 文档 | 仅当前消息，瞬时 |

- **@ 了文档** → AI 只检索被 @ 的文档，无视会话绑定的知识库
- **没 @ 文档** → 沿用会话绑定的知识库（现有行为不变）

## 交互流程

### 触发与选择
- 输入框键入 `@` → 候选列表（来自会话绑定知识库的已索引文档）
- 可 @ 多个文档，支持搜索过滤和键盘选择

### 输入状态
- 选中文档以绿色 chip 显示
- 输入框下方显示辅助文字：`AI 将基于选中的文档回答`
- 可点 × 删除单个 chip，删除全部后恢复普通模式

### 发送后
- 用户消息气泡下方显示 `📄 基于：文档名`
- 标签保留到历史消息，回看时仍可见

### 未找到信息
- AI 回答"在这些文档中没有找到相关信息"，不静默回退到知识库

## 技术方案

### 后端

1. **接收层**：`chats.py` 把 `request_data.document_ids` 传给 `agent_service.process()`
2. **存储层**：用户消息 metadata 写入 `document_ids` 和 `document_names`
3. **Runtime**：`run()` 接收 `document_ids`，传入 `ToolContext`；系统提示追加焦点文档说明
4. **工具层**：`ToolContext` 新增 `document_ids` 字段；`search_documents` 和 `grep_documents` 强制按 `document_ids` 过滤

### 前端

1. `Message` 接口新增 `documentIds` 和 `documentNames`
2. 从 API 响应的 `metadata` 中提取并渲染
3. 用户消息气泡下方渲染 `📄 基于：...` 标签
4. 输入框下方显示 `AI 将基于选中的文档回答` 提示
5. `sendMessage` 时同步更新本地消息对象（无需等历史回读）
