# 智能分类方案重构：路径优先 + 多级嵌套

## Context

当前分类系统存在两个核心问题：

1. **路径分类过于粗糙** — 仅按第一级目录分组（`/docs/xxx` 全归入 `docs`），对于深度结构的文档站等于没分类。需求文档 13 设计的"自适应深度"算法从未实现。
2. **AI 分类不稳定** — LLM JSON 解析脆弱、批次合并粗糙、无质量校验、成本高。

用户提出改进方向：**先找公共父路径，剥离后再分类；分类支持多级嵌套，而非单层扁平**。在此基础上，设计路径优先 + AI 兜底的混合策略。

## 设计决策

### 1. 嵌套分类数据结构

`categories_json` 从扁平格式升级为递归嵌套：

```json
[
  {
    "category": "SDK",
    "pages": [],
    "children": [
      {
        "category": "iOS",
        "pages": [
          {"path": "/docs/sdk/ios/ui", "title": "UI Guide", "description": ""},
          {"path": "/docs/sdk/ios/network", "title": "Networking", "description": ""}
        ],
        "children": []
      },
      {
        "category": "Android",
        "pages": [...],
        "children": []
      }
    ]
  },
  {
    "category": "Guide",
    "pages": [
      {"path": "/docs/guide/intro", "title": "Introduction", "description": ""},
      {"path": "/docs/guide/advanced", "title": "Advanced Topics", "description": ""}
    ],
    "children": []
  }
]
```

节点规则：
- `category`: 分组名（路径段或 AI 生成的语义名）
- `pages`: 直接属于此节点的文档列表（叶节点有 pages，中间节点通常 pages=[] 但允许有）
- `children`: 子分组列表（叶节点 children=[]）
- 嵌套深度上限 3 级
- 旧格式不兼容 — 用户会删除重建所有知识库，无需前端自动包装或数据库迁移

### 2. 分类策略：auto / path_only / ai_only

`categorize_mode` 选项调整：

| 模式 | 行为 |
|------|------|
| `auto`（新默认） | 先路径分类 -> 评估质量 -> 不合格则 AI 兜底 |
| `path_only` | 仅路径分类，不调用 LLM |
| `ai_only` | 仅 AI 分类（现有 AI 逻辑适配嵌套输出） |
| `skip` | 不分类（保持不变） |

`path_prefix` 模式废弃（`auto` 已覆盖且更优），旧任务中 `path_prefix` 视为 `auto`。

### 3. 路径分类算法

#### 3.1 找公共前缀

```
所有路径取最长公共路径前缀（LCP）
例: /docs/sdk/ios/ui, /docs/sdk/android/ui -> LCP = /docs/sdk/
例: /guide/intro, /api/ref, /blog/post -> LCP = /
剥离公共前缀后的"相对路径"用于后续分组
```

#### 3.2 构建 Trie 树

从相对路径构建 trie，每个节点记录经过它的文档列表。

#### 3.3 自适应深度裁剪

遍历 trie，确定每个子树的"最佳分组深度"：
- 如果某节点下文档数 >= `min_group_size`（默认 3），且有多个子分支 -> 该节点成为分类
- 如果某节点下文档数 < `min_group_size` -> 提升到父节点
- 如果某节点只有 1 个子分支且文档数少 -> 沿路径合并（如 `sdk/ios/ui` 合并为一个分类而非三级）

#### 3.4 质量评估（auto 模式用）

路径分类完成后评估质量：
- **覆盖率**：非"Other"页面占比 >= 70%
- **组数合理性**：3-15 组
- **均匀度**：最大组占比 < 50%

三个维度全部合格 -> 用路径结果；否则 -> AI 兜底。

### 4. AI 兜底策略（已确认）

auto 模式下路径分类不合格时，把路径分组结果作为 hint 传给 LLM，让其**微调优化**（合并过碎的、拆分过大的、优化命名）。不走"从零分类"路径。

AI 返回扁平结果，由后端包装为嵌套结构（更简单可靠）。

### 5. 前端嵌套树形 UI

侧边栏分类展示从扁平 accordion 改为递归树：
- 每个分类节点可展开/折叠
- 中间节点显示子分类列表
- 叶节点显示文档列表
- 层级通过缩进表达（每级约 16px indent）
- 复用现有 `expandedGroups` Set 状态管理，key 改为路径式（如 `"SDK/iOS"`）

## 实施步骤

### Step 1: 嵌套类型定义

**后端** `backend/models/dto.py`：
- 新增 `CategoryNode` 模型（递归）：`category: str`, `pages: list[SitemapPage]`, `children: list[CategoryNode]`

**前端** `frontend/src/utils/categoryParser.ts`：
- `CategoryGroup` 改为递归 `CategoryNode`：`category: string`, `pages: SitemapPage[]`, `children?: CategoryNode[]`
- 新增递归工具函数：`flattenCategories(nodes)` -> 扁平列表、`countPages(node)` -> 递归计数、`findCategoryForPath(nodes, path)` -> 递归查找

### Step 2: 路径分类算法重写

**文件** `backend/services/task_service.py`：

重写 `_build_path_groups` 为 `_build_nested_path_groups`：

```python
def _build_nested_path_groups(pages: list[dict], min_group_size: int = 3) -> list[dict]:
    """
    1. 找最长公共路径前缀（LCP），剥离
    2. 从相对路径构建 trie
    3. 自适应裁剪 trie -> 嵌套 CategoryNode 列表
    4. 小组合并到父节点
    """
```

新增辅助函数：
- `_longest_common_prefix(paths: list[str]) -> str` — 逐段比较
- `_build_trie(pages_with_rel_path)` — 构建 trie 字典
- `_prune_trie(trie, min_group_size)` — 裁剪 + 构建嵌套结构

### Step 3: 质量评估函数

**文件** `backend/services/task_service.py`：

新增 `_evaluate_path_quality(groups: list[dict], total_pages: int) -> tuple[bool, str]`：
- 计算覆盖率、组数、均匀度
- 返回 `(合格?, 原因说明)`

### Step 4: 混合策略调度

**文件** `backend/services/task_service.py`：

修改 `_categorize_collection`：
- `categorize_mode == "auto"` -> 调 `_build_nested_path_groups` -> `_evaluate_path_quality` -> 合格直接用 / 不合格调 AI 优化
- `categorize_mode == "path_only"` -> 直接 `_build_nested_path_groups`
- `categorize_mode == "ai_only"` -> AI 分类（适配嵌套输出）

### Step 5: AI 分类适配嵌套输出

**文件** `backend/services/llm_service.py`：

修改 `_categorize_pages_batch` prompt：
- 要求返回嵌套 JSON（带 `children`）
- 或者保持现有 AI 返回扁平结果，由后端包装为嵌套（更简单）

新增 `refine_categories(existing_groups, pages) -> list[dict]`：
- auto 模式 AI 兜底用
- prompt：给定路径分组结果，让 LLM 调整（合并过碎、拆分过大、命名优化）

### Step 6: README 生成适配嵌套

**文件** `backend/services/llm_service.py`：

修改 `_build_groups_text`：
- 递归遍历嵌套结构
- 用缩进表达层级：`## SDK` -> `### iOS` -> `- page1`

修改 `generate_readme_content` prompt：
- 说明输入是层级结构
- 生成多级标题（`##` / `###` / `####`）对应分类层级

### Step 7: 增量分类适配

**文件** `backend/services/task_service.py`：

修改 `_categorize_incremental`：
- `path_only` / `auto`（路径合格时）-> 全量重建（路径分类本身快，无需增量）
- `ai_only` / `auto`（AI 兜底时）-> 现有 merge + optimize 逻辑，适配嵌套结构

### Step 8: 前端侧边栏树形渲染

**文件** `frontend/src/components/knowledge/KnowledgeBaseManagement.tsx`：

替换扁平 accordion 为递归组件 `CategoryTree`：
- 新组件 `CategoryTreeNode`：接收一个 `CategoryNode`，递归渲染 children
- 每级缩进 `pl-4`（16px）
- 中间节点：chevron + 分类名 + 总文档数 badge -> 展开显示 children
- 叶节点：chevron + 分类名 + 文档数 badge -> 展开显示文档列表
- `expandedGroups` Set 的 key 改为路径式（如 `"SDK"`、`"SDK/iOS"`）

更新 `getDisplayCategory` / `categoryNameMap` 逻辑适配嵌套结构。

### Step 9: 前端分类解析器重写

**文件** `frontend/src/utils/categoryParser.ts`：

- `CategoryGroup` 改为递归 `CategoryNode`：`category: string`, `pages: SitemapPage[]`, `children?: CategoryNode[]`
- `parseCategories` 直接解析嵌套 JSON（无需兼容旧格式，用户会删除重建）
- 新增递归函数：`flattenCategories`, `countPages`, `findCategoryForPath`, `getCategoryNames` 全部改为递归版本

### Step 10: 请求/响应模型更新

**后端** `backend/models/requests.py`：
- `categorize_mode` 默认值从 `"ai"` 改为 `"auto"`

**前端** `frontend/src/components/InputDialog.tsx`：

1. **路径分类**：抓取一个有清晰路径结构的文档站 -> 按路径自动分出多级嵌套分类

4. **AI 兜底**：auto 模式下路径不合格 -> AI 分类成功、结果为嵌套格式
5. **前端树形 UI**：侧边栏显示多级可展开/折叠的分类树
6. **README**：生成的 README 包含多级标题对应分类层级
7. **增量分类**：新增文档后重新分类，嵌套结构正确更新
8. **回归**：现有测试通过
