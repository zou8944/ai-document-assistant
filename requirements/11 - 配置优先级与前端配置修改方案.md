# 配置集中化方案：全部收入 Settings 表

## 核心决策

所有配置项统一存放在数据库 `settings` 表中，不再通过环境变量读取任何业务配置。环境变量仅用于：

1. **启动服务本身的最小依赖**（如首次启动时连接数据库的默认路径，仅作为兜底）
2. **区分部署环境**（如 `ENV=development` / `ENV=production`）

除此之外，**一切可调配置均走 settings 表**。

## 为什么这么做

1. **配置来源唯一**：每个 key 只存在一个地方，彻底消除优先级歧义和维护混乱
2. **用户体验统一**：前端改什么，后端就用什么，不会被暗中覆盖
3. ** onboarding 更友好**：新用户首次打开应用时，无需提前准备 `.env` 文件，应用自动引导填写关键信息
4. **部署简化**：容器化部署时不再需要挂载 `.env` 或注入大量环境变量

## 配置分类与每项说明

每项配置在界面上均需附带用途说明，让用户知道它是干什么的。

| 分类 | 配置项 | 用途说明 | 默认值 | 前端是否可见 |
|---|---|---|---|---|
| **关键信息** | `OPENAI_API_KEY` | 调用 OpenAI 大模型所需的 API 密钥，不填写无法进行 AI 对话 | — | 可见，必填 |
| | `EMBEDDING_API_KEY` | 调用文本向量化模型所需的 API 密钥，用于将文档内容转为向量存储 | — | 可见，必填 |
| **业务参数** | `LLM_MODEL` | 对话时使用的 AI 模型名称，如 `gpt-4o` | `gpt-4o` | 可见，可改 |
| | `CHUNK_SIZE` | 文档切分时的每段最大字符数，决定检索时的引用片段长度 | `500` | 可见，可改 |
| | `CHUNK_OVERLAP` | 相邻文本块之间的重叠字符数，保证上下文衔接不丢失信息 | `50` | 可见，可改 |
| | `CRAWLER_MAX_DEPTH` | 从初始 URL 开始递归抓取网页的最大深度，越大抓得越多越慢 | `3` | 可见，可改 |
| | `RAG_TOP_K` | 回答问题时从知识库检索的最相似文档片段数量 | `5` | 可见，可改 |
| | `AGENT_TEMPERATURE` | AI 回复的随机程度，值越低越稳定，越高越有创造性 | `0.7` | 可见，可改 |
| **系统参数** | `SERVER_PORT` | 后端服务监听的端口 | `8000` | 可见（高级模式），可改 |
| | `LOG_LEVEL` | 服务端日志输出级别：`DEBUG` / `INFO` / `WARNING` / `ERROR` | `INFO` | 可见（高级模式），可改 |

## 配置完整性检测与引导

### 何时触发引导

只要检测到配置不完整（无论首次打开还是后续使用），即引导用户补全：

1. **应用启动时**：前端初始化阶段调用 `GET /api/settings/status`，后端返回关键信息配置状态
2. **路由切换时**：每次切换路由前检测，若配置不完整则拦截并跳转引导页
3. **功能调用时**：用户尝试使用依赖未配置项的功能（如发起对话但未配置 API Key），弹出引导提示

后端检测逻辑：遍历「关键信息」类配置，任一值为空即判定为不完整。

### 引导流程

1. 前端检测到配置不完整，自动跳转「配置引导」页，禁止进入主界面
2. 用户填写并保存后，后端校验并写入 `settings` 表
3. 全部关键信息配置完成后，自动进入主界面

## 后端读取逻辑

```python
# 统一入口，彻底摒弃 os.getenv() 读取业务配置
from backend.db import get_setting

model = get_setting("LLM_MODEL", default="gpt-4o")
api_key = get_setting("OPENAI_API_KEY")  # 无默认值，未设置时返回 None
```

唯一允许使用环境变量的地方：

```python
import os

# 仅用于判断部署环境或启动级参数
ENV = os.getenv("ENV", "development")
```

## 前端设置页设计

- **首次引导页**：简洁表单，仅展示「关键信息」类配置，必须全部填写才能进入主界面
- **设置页（常规）**：左侧分类导航，右侧表单
  - 常用：模型选择、CHUNK_SIZE、RAG_TOP_K 等
  - 高级：服务端端口、日志级别等（折叠或需二次确认展开）
- **保存即生效**：后端更新 `settings` 表后，相关模块热重载或下次调用时自动读取新值

## 数据安全

- `settings` 表中敏感字段（API Key）使用 AES 加密存储，密钥由服务启动时生成的或内置的根密钥派生
- 前端展示 API Key 时做脱敏处理（如 `sk-****xxxx`），仅在输入框中可见完整值

## 实施步骤

1. 定义 `settings` 表结构（含 `key`, `value`, `category`, `encrypted` 等字段）
2. 编写统一的 `get_setting()` / `set_setting()` 后端接口
3. 替换所有直接读取环境变量的业务代码，改为调用 `get_setting()`
4. 前端实现「首次引导」和「设置页」
5. 添加敏感字段加密/解密逻辑
6. 废弃 `.env` 中除 `ENV` 外的所有业务配置项，更新部署文档

## 结论

- **所有配置归一**：`settings` 表是唯一配置来源
- **环境变量退场**：不再作为业务配置读取途径
- **引导式 onboarding**：首次使用自动引导填写关键信息，降低上手门槛
