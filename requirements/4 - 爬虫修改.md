# 爬虫修改需求 - 静态镜像与离线可用

## 背景与结论

之前方案是 HTML 继续引用线上 CSS/JS/图片域名，这在实际落地中不可靠：

- 会触发同源策略限制，容易出现 CORS 错误
- 大量静态资源使用相对路径，脱离原站路由后解析会失效
- 不同页面共享公共资源（CSS/JS）是常态，无法只改页面链接而不处理资源

**最终结论**：爬虫输出必须是可离线访问的本地镜像。页面链接和静态资源链接都要重写到本地，静态资源必须下载。

---

## 需求整理

### 1. 爬取范围

- 遍历同域名页面（优先 `sitemap.xml`，缺失时降级 BFS 遍历 `<a>`）
- URL 规范化去重（query、trailing slash、大小写、fragment）
- 保持当前决策：暂不支持动态渲染站点（不引入 Playwright）
- 支持中断续爬：每个页面爬取前先查库，已成功入库页面直接跳过，不重复请求目标站点

### 2. 页面存储

- 每个页面至少保留两份内容：
  - 原始 HTML（用于还原展示）
  - Markdown（用于索引/RAG）
- Markdown 头部记录元信息：`source_url`、`title`、`crawled_at`
- 页面需保存原始 URL 与本地存储路径映射

### 3. 本地化目标（核心）

- HTML 中页面跳转链接改写为本地相对路径
- HTML 中静态资源链接也改写为本地相对路径，包括：
  - CSS（`<link href>`、CSS 内 `@import`）
  - JS（`<script src>`）
  - 图片（`<img src>`、`srcset`）
  - 字体、SVG、favicon、媒体文件（`url(...)`、`<source src>` 等）
- 对跨域但被页面引用的静态资源允许下载并本地化（不再限定“仅同域资源”）

### 4. 站点结构与可视化

- 爬取完成后生成站点结构（sitemap tree）
- 先用 URL 路径规则构建层级，再用 AI 补充描述和标签
- 可视化是 MVP 必选项（至少树形视图，后续可扩图形关系）

## 方案设计

### 总体阶段

```
Phase 1 - 增量爬取与落库（关键改造）
  ├── 页面级去重：爬取前先判断 DB 是否已存在且状态为 indexed
  ├── 边爬边入库：每成功/失败 1 页立刻写入，不等待整批结束
  ├── 任务中断后可续跑：下一次自动跳过已入库页面
  └── 同时产出 HTML + Markdown，并建立 URL -> 本地路径映射

Phase 2 - 静态资源镜像（新增核心）
  ├── 从 HTML/CSS 中提取全部静态资源 URL
  ├── 下载资源到本地 assets 目录（按 URL 哈希或规范路径落盘）
  ├── 资源去重（同 URL 只下载一次）
  └── 记录资源映射：remote_url -> local_path

Phase 3 - 全量链接重写
  ├── 重写页面链接 href 到本地 html
  ├── 重写静态资源链接到本地 assets
  ├── 处理相对路径、绝对路径、协议相对路径
  └── 重写后做可访问性校验（无本地 404）

Phase 4 - Sitemap 与可视化
  ├── URL 树 + AI 语义增强
  ├── 写入 collections.sitemap_json
  └── 前端树形/图形展示
```

### 关键实现点

0. 增量容错执行
- 不能采用“先批量爬取完再统一入库”的模式
- 正确流程：`discover url -> check db -> crawl one page -> persist one page -> crawl next`
- 判断规则建议：
  - `status = indexed`：视为已完成，直接跳过
  - `status = failed`：允许下一次重试

1. 资源提取器
- HTML：解析 `href/src/srcset/poster/data-*`
- CSS：解析 `url(...)` 与 `@import`
- 统一将相对路径基于页面 URL 解析成绝对 URL 再入队

2. 资源下载器
- 下载时保留内容类型和扩展名（无法识别时用二进制后缀策略）
- 对公共资源做全局去重，避免重复下载
- 支持重试与失败标记（失败资源可保留原链接作为降级）

3. 链接改写器
- 先构建两张映射表：
  - `page_url -> local_html_path`
  - `asset_url -> local_asset_path`
- 统一用相对路径写回 HTML/CSS，确保整个站点可搬迁

4. 一致性校验
- 对每个页面检查：
  - 本地页面链接命中率
  - 本地静态资源命中率
- 输出校验报告（失败清单 + 原因）

## 数据与存储建议

### 文件结构（建议）

```
cache/
└── docs.example.com/
    ├── pages/
    │   ├── guide/intro.html
    │   └── guide/intro.md
    ├── assets/
    │   ├── css/...
    │   ├── js/...
    │   ├── img/...
    │   └── fonts/...
    └── manifests/
        ├── pages.json      # page_url -> local_path
        └── assets.json     # asset_url -> local_path
```

### 数据库字段（沿用并补充）

- `documents.source_path`：页面原始路径（用于映射与重写）
- `documents.html_content`：重写后的 HTML（供前端预览）
- `collections.sitemap_json`：站点树与语义信息
- 可新增资源清单表（可选）：记录资源下载状态、类型、大小、失败原因

---

## 风险与边界

- 外链静态资源过多会拉长爬取时间，需要并发与限速控制
- 某些资源有防盗链/鉴权，可能无法下载；需降级策略
- 动态注入资源（运行时 JS 拼接）在不执行 JS 的前提下可能漏抓
- 站点规模大时，增量续爬依赖“逐页入库”的执行模型，禁止回退到批量入库模式

---

## 当前决策清单

1. 静态资源必须下载并本地化，不再依赖线上域名
2. 链接重写范围包含页面链接和静态资源链接，不仅是 `<a href>`
3. `sitemap.xml` 优先，缺失时 BFS 遍历
4. 暂不支持动态页面渲染，不引入 Playwright
5. Sitemap 可视化为 MVP 必选项
6. URL 任务采用“边爬边入库”，中断后续爬必须跳过 `indexed` 页面
