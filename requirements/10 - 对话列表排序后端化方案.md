# 对话列表排序后端化方案

## 背景与目标

当前侧边栏的对话列表（`chatSessions`）完全靠 [appStore.ts](frontend/src/store/appStore.ts) 的 `zustand/persist` 中间件写入 `localStorage`。这带来若干问题：

1. **拖动排序仅在本地生效**。后端 `chats` 表没有顺序字段，刷新后顺序回到「按 `last_message_at` 倒序」。
2. **重命名 `handleChatRename` 和删除 `handleChatDelete` 实际只改本地 store**（[Sidebar.tsx:92-100](frontend/src/components/layout/Sidebar.tsx#L92-L100)），没有同步到后端。一旦清缓存 / 换设备，这两项操作就丢失。
3. 启动时不从后端拉取列表，完全依赖本地缓存恢复。

本方案目标：

- **所有 chat 列表的变更操作（拖动排序、重命名、删除）都以后端为准**。
- 启动时由前端从后端拉取完整列表渲染。
- 从 `persist` 的 `partialize` 中移除 `chatSessions`，结束本地持久化。

## 不在本次范围

- `knowledgeBases` 的本地持久化（保留）。
- 多端同步、并发冲突解决（当前单用户场景）。
- 软删除 / 回收站。
- 拖到列表最末尾的视觉指示线问题（独立小修，不在本方案）。

## 后端设计

### 1. 数据库 schema

`chats` 表新增字段：

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `sort_order` | `Integer` | `NOT NULL`，默认 `0` | 列表显示顺序，越小越靠前 |

新增索引：`idx_chats_sort_order` 单列索引。

### 2. Alembic 迁移

新建 `backend/database/migrations/versions/<rev>_add_sort_order_to_chats.py`：

**upgrade**：

1. `op.add_column("chats", sa.Column("sort_order", sa.Integer(), nullable=False, server_default="0"))`
2. 回填现有数据 —— 用 SQLAlchemy Core 表达式（DB 无关）按当前 UI 顺序（`last_message_at DESC NULLS LAST, updated_at DESC`）赋值：

   ```python
   bind = op.get_bind()
   chats_t = sa.table(
       "chats",
       sa.column("id", sa.String),
       sa.column("sort_order", sa.Integer),
       sa.column("last_message_at", sa.DateTime),
       sa.column("updated_at", sa.DateTime),
   )
   rows = bind.execute(
       sa.select(chats_t.c.id).order_by(
           chats_t.c.last_message_at.desc().nullslast(),
           chats_t.c.updated_at.desc(),
       )
   ).scalars().all()
   for pos, chat_id in enumerate(rows):
       bind.execute(
           chats_t.update().where(chats_t.c.id == chat_id).values(sort_order=pos)
       )
   ```

3. `op.create_index("idx_chats_sort_order", "chats", ["sort_order"])`

**downgrade**：

1. `op.drop_index("idx_chats_sort_order", table_name="chats")`
2. `op.drop_column("chats", "sort_order")`

### 3. SQLAlchemy 模型

[backend/database/models/chat.py](backend/database/models/chat.py) 的 `Chat` 加：

```python
sort_order: Mapped[int] = mapped_column(
    Integer,
    nullable=False,
    default=0,
    server_default="0",
    doc="Display order in chat list (smaller comes first)",
)
```

并在 `__table_args__` 中加入 `Index("idx_chats_sort_order", "sort_order")`。

### 4. DTO

`backend/models/dto.py` 中 `ChatDTO` 新增 `sort_order: int` 字段。

### 5. Repository

[backend/repository/chat.py](backend/repository/chat.py) 的 `ChatRepository` 改动：

- `get_all_ordered`：`order_by` 改为 `Chat.sort_order.asc()`，移除原 `last_message_at` 排序。
- 新增 `next_sort_order() -> int`：

  ```python
  def next_sort_order(self) -> int:
      with session_context() as session:
          max_order = session.scalar(select(func.max(Chat.sort_order)))
          return (max_order if max_order is not None else -1) + 1
  ```

- 新增 `reorder(ordered_ids: list[str]) -> int`：单事务里按位置写入：

  ```python
  def reorder(self, ordered_ids: list[str]) -> int:
      with session_context() as session:
          chats_map = {
              c.id: c
              for c in session.scalars(
                  select(Chat).where(Chat.id.in_(ordered_ids))
              )
          }
          if len(chats_map) != len(ordered_ids):
              missing = set(ordered_ids) - chats_map.keys()
              raise ValueError(f"Chats not found: {missing}")
          for pos, chat_id in enumerate(ordered_ids):
              chats_map[chat_id].sort_order = pos
          return len(ordered_ids)
  ```

### 6. Service

[backend/services/chat_service.py](backend/services/chat_service.py) 新增 `reorder_chats(ordered_ids)`：

- 校验 `len(ordered_ids) == total_chat_count`（严格模式：必须提交完整列表）。
- 校验 ids 唯一（去重检查）。
- 调用 `ChatRepository.reorder`。
- 返回影响行数。

`create_chat` 改动：写入前调用 `next_sort_order` 给新 chat 落在末尾（保留现有 UX —— 新建对话出现在列表底部）。

### 7. 路由

[backend/api/routes/chats.py](backend/api/routes/chats.py) 新增：

```python
class ReorderChatsRequest(BaseModel):
    chat_ids: list[str]

@router.post("/chats/reorder")
async def reorder_chats(
    request_data: ReorderChatsRequest,
    request: Request,
):
    chat_service = get_app_state(request).chat_service
    try:
        count = await chat_service.reorder_chats(request_data.chat_ids)
    except ValueError as e:
        raise HTTPBadRequestException(str(e))
    return {"reordered": count}
```

`list_chats` 响应顺序由 repo 的新 order_by 决定，无需改路由。

## 前端设计

### 1. Store 调整 [appStore.ts](frontend/src/store/appStore.ts)

- `partialize` 中**移除** `chatSessions`：

  ```ts
  partialize: (state) => ({
    knowledgeBases: state.knowledgeBases,
    settings: state.settings,
    sidebarWidth: state.sidebarWidth,
    docChatSidebarOpen: state.docChatSidebarOpen,
    docChatSidebarWidth: state.docChatSidebarWidth,
    displayLanguage: state.displayLanguage,
  })
  ```

- 新增 action `setChatSessions(chats: ChatSession[])`：整批替换。

### 2. apiClient 新方法 [apiClient.ts](frontend/src/services/apiClient.ts)

```ts
async reorderChats(chatIds: string[]): Promise<APIResponse<{ reordered: number }>> {
  return this.request(`/api/v1/chats/reorder`, {
    method: 'POST',
    body: JSON.stringify({ chat_ids: chatIds }),
  })
}
```

### 3. 启动加载

在 `App.tsx` 中加 `useEffect`（或独立成 `useBootstrapChats` hook）：

```ts
useEffect(() => {
  apiClient.listChats()
    .then(res => {
      const data = extractData(res)
      const sessions = data.chats.map(mapChatDtoToSession)
      setChatSessions(sessions)
    })
    .catch(err => {
      console.error('Failed to load chats:', err)
      // toast 提示
    })
}, [])
```

`mapChatDtoToSession` 用来把后端 `Chat` 形状转换成前端 `ChatSession` 形状，逻辑与现有 `addChatSession` 处的映射一致。

### 4. Sidebar 操作改造 [Sidebar.tsx](frontend/src/components/layout/Sidebar.tsx)

**重命名**：

```ts
const handleChatRename = async (chatId: string, newName: string) => {
  try {
    await apiClient.updateChat(chatId, { name: newName })
    updateChatSession(chatId, { name: newName })
  } catch (err) {
    console.error('Rename failed:', err)
    alert('重命名失败: ' + (err as Error).message)
  }
}
```

**删除**：

```ts
const handleChatDelete = async (chatId: string) => {
  if (!window.confirm('确定要删除这个聊天吗？这个操作无法撤销。')) return
  try {
    await apiClient.deleteChat(chatId)
    deleteChatSession(chatId)
  } catch (err) {
    console.error('Delete failed:', err)
    alert('删除失败: ' + (err as Error).message)
  }
}
```

**拖动排序**（核心）：

```ts
const handleDrop = async () => {
  if (dragIndex === null || dragOverIndex === null || dragIndex === dragOverIndex) {
    return
  }
  // 1. 乐观更新
  reorderChatSessions(dragIndex, dragOverIndex)
  // 2. 取最新顺序提交后端
  const ids = useAppStore.getState().chatSessions.map(c => c.id)
  try {
    await apiClient.reorderChats(ids)
  } catch (err) {
    console.error('Reorder failed:', err)
    // 失败回滚：从后端重新拉
    const res = await apiClient.listChats()
    const data = extractData(res)
    setChatSessions(data.chats.map(mapChatDtoToSession))
    alert('排序失败，已恢复到服务端最新状态')
  }
}
```

### 5. 数据流

```
启动:
  App.mount → apiClient.listChats() → store.setChatSessions(...)

拖动结束:
  Sidebar.handleDrop:
    ├─ store.reorderChatSessions(from, to)  (乐观)
    ├─ ids = store.chatSessions.map(c => c.id)
    └─ apiClient.reorderChats(ids)
         ├─ 成功 → 完成
         └─ 失败 → apiClient.listChats() → store.setChatSessions(...) → toast

重命名 / 删除:
  apiClient.updateChat / deleteChat
    ├─ 成功 → store.updateChatSession / deleteChatSession
    └─ 失败 → toast，不改本地状态
```

## 错误处理原则

- **网络 / 服务端失败一律：toast 报错 + `listChats()` 重新拉取**，让前端 store 严格等于后端。
- 不在前端维护「旧顺序快照」用于回滚，简化代码、避免不一致。

## 兼容性与回滚

- **数据库**：迁移可双向（upgrade / downgrade），新增字段非破坏性。
- **API**：仅新增 `POST /chats/reorder`，对老前端无影响。`GET /chats` 排序变化为 `sort_order ASC`，老前端不感知（它本来不调用 listChats）。
- **前端**：升级后用户本地 `localStorage` 中的 `chatSessions` 会被忽略；首次启动后由后端列表替代。如果后端列表为空（全新部署），用户体验是空列表，需要重新创建对话 —— 但旧数据本就只在 localStorage，正是本方案要解决的问题。
