> 本次定位：**第二轮回合**，聚焦 #14 未覆盖的可访问性、设计令牌收尾、布局响应式、交互细节

## 一、Executive Summary

**Overall Score: 78 / 100**（#14 后 +16）

#14 的三个 P0 改造（Toast、统一 Modal、设计令牌）已落地，整体观感和 macOS 原生质感明显改善。但**第二层问题**浮现：

- **可访问性硬伤**：CJK 用户 IME 必现误触发送、6 处图标按钮零 `aria-label`、触摸目标 ~24px 远低于 44pt 阈值
- **设计令牌收尾不彻底**：22464ac 重构后仍有 ~150+ 处硬编码 `text-[#xxx]`、`bg-[#xxx]` 任意值
- **交互细节缺位**：侧边栏拖拽无宽度钳制 / 无键盘支持、AI 消息无 copy & regenerate、ResizableHandle 仅 1px 命中区

**Top 5 Problems:**

1. **CJK 用户按 Enter 误触发送** — 在中文/日文/韩文输入法选词时按 Enter 提交候选词，会立即发送尚未写完的问题
2. **6+ 处图标按钮零 `aria-label`** — 仅 `title` 属性，屏幕阅读器不读，键盘用户无法聚焦
3. **触摸目标远低于 44pt** — `p-1` 包裹 16px 图标 ≈ 24×24px，违反 macOS HIG
4. **设计令牌绕过** — 大量 `text-[#1c1c1e]`/`bg-[#007AFF]/10` 等本可直接用 `text-ink`/`bg-accent/10`
5. **侧边栏拖拽无宽度钳制 / 无键盘 / 无 ARIA role** — `ResizableHandle` 体验与无障碍双重缺失

## 二、Issues 清单

### Issue 1: CJK 用户 IME 选词回车误触发送

**Priority: P0**
**Category: Accessibility / i18n**

**Severity Reason:** 中文/日文/韩文用户在输入法候选词浮窗上按 Enter 确认选词时，会被当成"发送"立即发出未完成的句子。这是 #14 期间被遗漏的 i18n 硬伤。

**Current Behavior:**

- [ChatInterface.tsx:371-376](frontend/src/components/chat/ChatInterface.tsx#L371-L376) `onKeyDown` 仅判 `shiftKey`，未判 IME 合成状态
- [RichTextInput.tsx:146-158](frontend/src/components/chat/RichTextInput.tsx#L146-L158) 同上
- 复现：输入"你好世"→ 按 Enter 选"界" → 触发发送，实际只发出"你好世"

**Expected Behavior:**

- 发送前检查 `e.nativeEvent.isComposing === true` 或 `e.keyCode === 229`（兼容旧浏览器）
- 任一为真则不发送、不 blur 输入框

**Recommendation:** 🟢 Low Cost（1 小时）

```ts
const handleKeyDown = (e: React.KeyboardEvent<HTMLTextAreaElement>) => {
  // IME 合成中：让候选词确认走，不触发发送
  if (e.nativeEvent.isComposing || e.keyCode === 229) return
  if (e.key === 'Enter' && !e.shiftKey) { ... }
}
```

**Current Behavior（仅 `title`）:**

- [Sidebar.tsx:271-277](frontend/src/components/layout/Sidebar.tsx#L271-L277) — 新建对话 `+` 按钮
- [ChatItem.tsx:201-211](frontend/src/components/layout/ChatItem.tsx#L201-L211) — kebab 菜单
- [KnowledgeBaseOverview.tsx:412-415](frontend/src/components/knowledge/KnowledgeBaseOverview.tsx#L412-L415) — KB 卡片 kebab

- [FileUpload.tsx:187-191](frontend/src/components/FileUpload.tsx#L187-L191) — 文件 remove × 按钮

**Expected Behavior:** 每处加 `aria-label="..."`（中文即可，与现有中文 UI 一致）。

**Recommendation:** 🟢 Low Cost（1-2 小时）

### Issue 3: 触摸目标远低于 44×44 pt 阈值

**Priority: P0**
**Category: Accessibility / macOS HIG**

**Severity Reason:** macOS HIG 与 WCAG 2.5.5（AAA）均强制 ≥ 44×44 pt。当前 `p-1` + 16px 图标 ≈ 24×24 px，键盘 / 切换控制 / 触屏用户均无法操作。

**Current Behavior:** 见 Issue 2 列表中 6 处均使用 `p-1` + `w-4 h-4`。另：

- [Modal.tsx:131-134](frontend/src/components/common/Modal.tsx#L131-L134) — 关闭 ×
- [ToastContainer.tsx:78-84](frontend/src/components/common/ToastContainer.tsx#L78-L84) — Toast 关闭

**Expected Behavior:** 包裹 `inline-flex items-center justify-center min-h-[44px] min-w-[44px]`，或视觉保持 16px 但加 `hitSlop` 扩展命中区。

**Recommendation:** 🟢 Low Cost（1-2 小时）

### Issue 5: 侧边栏拖拽手柄无宽度钳制 / 无键盘 / 无 ARIA

**Priority: P1**
**Category: Layout / Accessibility**

**Severity Reason:** `ResizableHandle` 既是布局缺陷（拉到极窄/极宽破坏布局），也是无障碍缺口（无 `role="separator"`、无键盘支持）。

**Current Behavior:**

- [MainLayout.tsx:45-47](frontend/src/components/layout/MainLayout.tsx#L45-L47) — `setSidebarWidth(sidebarWidth + deltaX)` 无钳制
- [ResizableHandle.tsx](frontend/src/components/layout/ResizableHandle.tsx) — 1px 可见，鼠标命中靠 `-left-1` 扩展但无 ARIA

**Expected Behavior:**

- 钳制 `MIN=200, MAX=480` 像素，store 持久化
- 元素加 `role="separator" aria-orientation="vertical" aria-valuenow={width} aria-valuemin={200} aria-valuemax={480} tabIndex={0}`
- 键盘：聚焦后 ← / → 每次 ±16px 调整

**Recommendation:** 🟡 Medium Cost（半天）

### Issue 6: AI 消息无 copy / regenerate 工具栏

- 👍 / 👎 反馈（可选 Phase 2）

**Recommendation:** 🟡 Medium Cost（半天）

**Recommendation:** 🟢 Low Cost（1-2 小时）

## 四、验证策略

每个 commit 完成后：

1. **lint + type-check**：`cd frontend && npm run lint && npm run typecheck`
2. **测试**：`npm test`（已有 Vitest 用例需全绿）
3. **手动验证**：
   - IME：中文输入"你好世"按 Enter 不应发送
   - a11y：开启 macOS VoiceOver / 键盘 Tab 顺序合理
   - 触摸目标：在小窗口模拟器测 16px 图标可达区域
   - 拖拽：钳制 200/480 边界、←/→ 键盘调宽度
4. **视觉回归**：核心页（Chat、KB 概览、设置）截图对比
