# 14 - 前端 UX 审查与一致性改造方案

> 审查范围：整个 `frontend/` 项目
> 审查基准：[UI 设计指南.md](../UI%20设计指南.md)（Apple Liquid Glass + macOS HIG）
> 审查方法：对照设计指南、阅读核心组件源码、列出可执行改造任务
> 落地形态：12 个独立 commit，每 commit 解决 1-2 个 P0/P1/P2 问题

---

## 一、Executive Summary

**Overall Score: 62 / 100**

应用整体呈现明确的 macOS HIG / Liquid Glass 设计意图，但在多个关键维度上**设计语言执行不一致**，存在设计指南要求的 `#007AFF` 与 Tailwind 蓝混用、Glass Modal 与旧式 Modal 并存、错误反馈靠 `window.alert()` 等结构性偏差。新用户进入时的"零到上手"路径存在断点，**无新用户引导**、无空状态 CTA、一致性缺口显著。

**Top 3 Problems:**

1. **设计语言执行不一致**：UI 指南要求 `#007AFF` 系统蓝、`#1c1c1e` 主文字、统一半透明毛玻璃，但代码中混用 `bg-blue-500/600`、`text-gray-900`、`text-gray-600` 与指南色值，至少 3 种模态样式并存
2. **错误反馈原始且不 macOS 化**：全站关键错误用 `window.alert()` 弹窗，违反了"柔和、克制"的指南气质；`macos-toast` 样式已定义却未使用
3. **零状态体验缺失**：首次打开应用、首次创建对话、首次进入设置等多处空状态没有引导性 CTA，新用户不知道下一步该做什么

## 二、User Journey Analysis

### 核心路径 A：首次打开 → 第一个问答

**Current Flow:**

1. 启动后看到 `StartupScreen`（"正在启动应用..."）
2. 后端就绪 → 进入 `SetupWizard`（填 4 组 API 配置）
3. 填写完毕 → 进入 `KnowledgeBaseOverview`，看到空状态插画 + "创建知识库"按钮
4. 点击"创建知识库" → 模态弹窗输入名称描述 → 关闭
5. 进入 `KnowledgeBaseManagement` → 点击导入按钮 → 选择文件/URL → 等待任务
6. 任务完成 → 返回 `KnowledgeBaseOverview` → 点击知识库卡片的"聊天"按钮 → 创建/打开 chat
7. 在 chat 中输入问题 → 等待 AI 回答
8. 看到回答 + 思考过程 + 引用来源

**Ideal Flow:**

1. 启动 → 1 步完成（无感知完成 setup，因为配置项可推迟）
2. 第一次进入看到 3 个示例知识库或示例对话（沙箱数据）
3. 直接拖入文件 → 看到任务进度 → 进入 chat

**关键缺口：**

- 步骤 4 后，从"创建知识库"到"进入管理界面"没有任何过渡指引，用户不知道接下来是导入文档
- 步骤 5 的"导入文件"按钮在折叠面板中（`importCollapsed` 默认 `true`），用户必须先意识到要展开
- 步骤 6 之前，用户从未被引导到 chat 路径

### 核心路径 B：日常使用 → 快速问答

**Current Flow:**

1. 启动 → chat 列表显示
2. 点击现有 chat → 看到历史消息
3. 在底部输入框输入
4. 发送

**问题点：**

- 列表中没有"知识库"或"最后消息时间"的视觉提示
- 切换到 chat 时没有动画/过渡（页面瞬切）
- 知识库与 chat 的绑定关系不可见（用户必须点击 chat 才能看到它绑定的 KB）
- 没有键盘快捷键

## 三、Issues 清单

### Issue 1: 设计语言执行混乱 — 蓝/灰/圆角/边框全部"半统一"

**Priority: P0**
**Category: UI / Consistency**

**Severity Reason:**
设计指南是项目的硬性规范，混用导致整个产品的"原生质感"打折，是首次打开就感知的视觉信任问题。

**Current Behavior:**

- 蓝色：3 种并存
  - `#007AFF`（Apple 系统蓝，指南要求）：`Sidebar.tsx:246`、`ChatInterface.tsx:384` 等显式用法
  - `bg-blue-500` / `bg-blue-600`（Tailwind 默认蓝）：`KnowledgeBaseOverview.tsx:318, 382, 483, 562, 619`
  - `bg-[#0066D6]`（亮蓝 hover）：`ChatInterface.tsx:393`
- 文字色：4 种并存
  - `text-[#1c1c1e]`（指南要求深灰黑）
  - `text-gray-900` / `text-gray-700` / `text-gray-600` / `text-gray-500`（Tailwind 默认）
  - `text-[#8E8E93]`（指南辅助色）✅
- 圆角：`rounded-md` (6px) / `rounded-lg` (8px) / `rounded-xl` (12px) / `rounded-2xl` (16px) 在同一类元素上并存
- 边框：`border-gray-200/40` / `border-gray-200/50` / `border-gray-200/60` / `border-white/20` / `border-white/30` / `border-white/40`（透明度 4 种）

**Expected Behavior:**
所有视觉令牌收敛到指南定义的一组值，全站替换。

**Root Cause:**

- 缺乏设计令牌（design tokens）集中管理
- Tailwind config 中虽有 `accent: '#007AFF'` 等定义，但开发者倾向直接用 Tailwind 默认色
- 没有 ESLint/Prettier 规则限制颜色用法

**Recommendation:** 🟢 Low Cost（1 天）

1. 在 `tailwind.config.js` 中删除 `macos-blue` 等冗余别名，统一使用 `accent`
2. 在 `globals.css` 的 `@layer base` 添加 `*, *::before, *::after` 兜底，限制 Tailwind 调色板
3. 全站搜索替换：
   - `bg-blue-500` → `bg-accent`
   - `bg-blue-600` → `bg-accent-hover`
   - `text-gray-900` → `text-ink`
   - `text-gray-700/600/500` → `text-ink/85`、`text-ink/70`、`text-ink/55`（透明度阶梯）
4. 统一圆角：列表/卡片用 `rounded-2xl`，按钮用 `rounded-xl`，小 chip 用 `rounded-md`
5. 统一边框透明度到 2 个档位：`border-white/20` 和 `border-white/40`

### Issue 2: 模态框样式不统一 — Glass 模态与旧式 Tailwind 模态并存

**Priority: P0**
**Category: UI / Consistency**

**Severity Reason:**
模态是用户与系统交互的关键节点，3 种模态样式让用户每次都重新适应；且旧式模态违反设计指南的"半透明毛玻璃"要求。

**Current Behavior:**

样式 A（指南要求 / Liquid Glass）：

- `KnowledgeBaseOverview.tsx:518-568` 编辑模态、`KnowledgeBaseOverview.tsx:572-626` 重启任务模态
- 特征：`bg-black/40 backdrop-blur-sm`，居中

样式 B（旧 Tailwind）：

- `AddKnowledgeBaseModal.tsx:88-112` 创建知识库
- `KnowledgeBaseSelector.tsx:98-122` 选择知识库
- 特征：`bg-gray-500 bg-opacity-75`、底部对齐 (`items-end`)、`inline-block align-bottom` 的古老 Tailwind 模式

样式 C（部分 Glass）：

- `FileUploadModal.tsx:39-56` 文件上传
- 特征：`glass-morph` + `bg-white/90 backdrop-blur-xl`

**Expected Behavior:**
全站模态统一为样式 A 的"半透明背景 + 居中 + Glass Panel"。

**Root Cause:**
历史遗留 — 模态组件在不同时期编写，未做收敛。

**Recommendation:** 🟡 Medium Cost（2-3 天）

1. 抽取一个 `<Modal>` 基础组件到 `src/components/common/Modal.tsx`，定义统一 prop：`title` / `children` / `onClose` / `footer`
2. 重写 `AddKnowledgeBaseModal`、`KnowledgeBaseSelector`、`FileUploadModal` 全部基于新 `<Modal>`
3. 把"创建知识库"和"重启任务"两个模态也从内联 JSX 改为复用 `<Modal>`

### Issue 3: 全站用 `window.alert()` 做错误反馈

**Priority: P0**
**Category: Error Handling / UI**

**Severity Reason:**
`alert()` 是阻塞的、视觉突兀的、不可定制的浏览器原生弹窗，严重破坏 macOS 原生质感；指南要求"柔和渐变"和"反馈及时"。

**Current Behavior:**

`alert()` 出现位置（最少 10+ 处）：

- `Sidebar.tsx:120, 142, 184, 227`（创建/重命名/删除/排序失败）
- `ChatInterface.tsx:152`（更新知识库失败）
- `KnowledgeBaseOverview.tsx:132, 198, 232, 274`（重启/重索引/创建聊天/编辑失败）
- `KnowledgeBaseManagement.tsx:427, 445, 463, 476, 510, 524, 549, 564, 575, 607, 625, 633, 643`（10+ 处）

**Expected Behavior:**

- 失败：用 Glass Toast 浮在右下角（指南样式的 `macos-toast.error`）
- 表单错误：行内红字提示
- 阻塞性确认（如删除）：用 Glass 确认弹窗

**Root Cause:**

- 没有全局通知/Toast 系统
- `globals.css` 已定义 `.macos-toast` 样式，但未配套 React 组件

**Recommendation:** 🟡 Medium Cost（1-2 天）

1. 创建 `src/components/common/Toast.tsx` + Zustand `useToastStore`
2. 替换全站 `alert(error.message)` 为 `toast.error('...')`
3. 把"删除确认"、"清空知识库确认"等高风险操作改为 `<ConfirmDialog>`（基于 Issue 2 的 Modal）

### Issue 4: 新用户零引导 — 启动后无 onboarding

**Priority: P0**
**Category: Onboarding / Information Architecture**

**Severity Reason:**
首次打开应用的用户面对一个空 sidebar + 空知识库概览，没有"30 秒内理解这是什么 / 能做什么 / 怎么做"的引导。

**Current Behavior:**

- `App.tsx`：`SetupWizard` → `KnowledgeBaseOverview`（空状态有插画 + "创建知识库"按钮 ✅）

- `ChatInterface` 空状态（`ChatInterface.tsx:187-196`）：只显示"请选择一个聊天会话"+ CPU icon，未引导用户去 sidebar 创建

**Expected Behavior:**

- 首次完成 setup 后，show 一次性 3 步引导（popover / coach mark）：①创建知识库 → ②导入文档 → ③发起对话
- 当 chat 列表为空时，主区域显示"创建你的第一个对话"卡片 + 大号 CTA 按钮

**Root Cause:**

- 没有 onboarding 状态管理（Zustand 里也没有 `hasSeenOnboarding`）

**Recommendation:** 🟡 Medium Cost（3-5 天）

2. 抽 `<CoachMark>` 组件（基于 floating-ui 或自实现）
3. 在 setup 完成后，3 个目标元素（"知识库"按钮、"添加知识库"按钮、聊天输入框）依次显示 popover

**Severity Reason:**
RichTextInput 支持 @ 引用特定文档（`RichTextInput.tsx`）— 但该功能完全隐藏在用户必须**先键入 @** 才能看到的选择器后面。新用户永远不知道。

## 七、总结
